Thema 01

Login-Strategien für Express hinter nginx

In einem typischen Setup leitet nginx alle eingehenden Anfragen an Express weiter. Die Authentifizierung kann dabei an verschiedenen Stellen ansetzen.

1 In Express

Flexibelste Option. nginx leitet durch, Express übernimmt Login und Session-Verwaltung selbst. Unterstützt lokale Nutzer, OAuth und JWT.

2 In nginx

Über auth_basic oder auth_request. Express bekommt nur authentifizierte Anfragen. Gut für interne Tools und Multi-Service-Setups.

3 OAuth / SSO

Login via Google, GitHub, Microsoft. Passport.js-Strategie handelt den OAuth-Flow ab; kein eigenes Passwort-Handling nötig.

Entscheidungsmatrix

SzenarioEmpfehlung
Klassische Web-AppSession + Passport (Local Strategy)
REST-API / SPAJWT im HttpOnly-Cookie
Interne Tools schnell absichernnginx Basic Auth
Mehrere Dienste zentralnginx auth_request
Soziales LoginOAuth 2.0 via Passport
nginx-Pflicht: Damit secure-Cookies hinter dem Proxy funktionieren, muss proxy_set_header X-Forwarded-Proto $scheme gesetzt und in Express app.set('trust proxy', 1) aktiviert sein.

Session-Login – Code erklärt

Das Session-Verfahren speichert den eingeloggten Nutzer serverseitig. Der Browser erhält nur eine bedeutungslose Session-ID als Cookie.

Session-Middleware

OptionBedeutung
secretSigniert die Session-ID – verhindert Fälschungen
resave: falseNur speichern bei Änderung
saveUninitialized: falseKeine leere Session für Gäste
httpOnly: trueCookie für JS unsichtbar → schützt vor XSS
sameSite: strictKein Cross-Site-Mitsendem → CSRF-Schutz
secure: trueNur über HTTPS übertragen

Serialize / Deserialize

Nach dem Login speichert serializeUser() nur die Nutzer-ID in der Session. Bei jedem Folge-Request lädt deserializeUser() den aktuellen Datensatz aus der DB. So wirken Änderungen (z. B. Kontosperrung) sofort.

Authentifizierungsablauf

POST /login (username + password) │ ▼ passport.authenticate() │ ▼ LocalStrategy DB-Lookup → bcrypt.compare() │ ├─FALSCH─▶ /login?error=1 │ └─RICHTIG─▶ serializeUser() │ ▼ Session anlegen Set-Cookie: id=abc123 │ ▼ /dashboard Folge-Request mit Cookie │ ▼ deserializeUser() → req.user = {…} │ ▼ Route-Handler ausgeführt
Produktions-Hinweis: express-session speichert Sessions standardmäßig im RAM. Für Produktivbetrieb einen persistenten Store verwenden: connect-redis oder connect-pg-simple.

Bearer Token – Einordnung

Bearer Token ist kein eigenständiges Protokoll, sondern ein Transportmechanismus (RFC 6750), ursprünglich Teil von OAuth 2.0, aber davon unabhängig einsetzbar. 

// HTTP-Header mit Bearer Token
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9…

Taxonomie

Token-basierte Authentifizierung │ ├─ Opaque Tokens │ Zufallsstring, serverseitig │ validiert (Datenbank-Lookup) │ └─ Self-contained Tokens Tragen alle Infos selbst │ └─ JWT ← häufigste Umsetzung Header · Payload · Signatur

Token-Verfahren im Vergleich

VerfahrenIdentitätsnachweis
Bearer TokenBloßer Besitz reicht
MAC TokenToken + Signatur der Request-Daten
mTLSClient-Zertifikat + Token
DPoPNachweis des privaten Schlüssels

Einsatzgebiete

A OAuth 2.0 / OIDC

Bearer Token als Access Token nach Authorization Code, Client Credentials oder Device Flow.

RFC 6750RFC 6749

B Stateless API (JWT)

Signiertes JWT als Bearer Token. Server prüft die Signatur lokal – kein Session-Store nötig.

HS256RS256

C Statische API-Keys

Vereinfachte Form: ein fixer Token-String, z. B. bei Stripe oder GitHub. Kein Ablaufdatum.

API-Key

Sicherheit: Bearer Token ist das einfachste, aber auch schwächste Token-Verfahren – wer das Token abfängt, hat vollen Zugriff. HTTPS ist daher zwingend. Die eigentliche Sicherheit ergibt sich aus dem Inhalt (JWT-Signatur) oder dem Ausstellungsverfahren (OAuth-Flow).
