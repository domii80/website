Praktisches Software-Testing

"Vertrauen ist gut, Testen ist besser."

Die Test-Pyramide: Das Fundament der Qualität

Die meisten Tests sollten unten sein (schnell & günstig), die wenigsten oben (langsam & teuer).

3. End-to-End (E2E) Tests (Die Kundenperspektive)

  • Was? Simuliert einen kompletten Benutzer-Workflow durch die gesamte Anwendung (Frontend -> Backend -> DB).
  • Merksatz: "Der Weg ist das Ziel – und er muss funktionieren."
  • Beispiel (Node.js/Playwright): "Mit Playwright einen kompletten Kaufprozess simulieren: User loggt sich ein, legt ein Produkt in den Warenkorb, geht zur Kasse und bestätigt die Bestellung."

2. Integrations-Tests (Das Zusammenspiel)

  • Was? Prüft, ob verschiedene Module oder Services korrekt miteinander kommunizieren.
  • Merksatz: "Die Kette ist nur so stark wie ihr schwächstes Glied."
  • Beispiel (Node.js/Express): "Ein Test, der einen Express API-Endpunkt aufruft und prüft, ob die korrekten Daten aus der Datenbank geholt und als JSON zurückgegeben werden, ohne das Frontend zu nutzen."

1. Unit-Tests (Die Bausteine)

  • Was? Testet die kleinste logische Einheit (eine Funktion, eine Methode) komplett isoliert.
  • Merksatz: "Viele kleine Bausteine ergeben eine stabile Mauer."
  • Beispiel (TypeScript): "Eine einzelne TypeScript-Funktion testen, z.B. calculateTotalPrice(items). expect(calculateTotalPrice([...])).toBe(19.99)."

Weitere Wichtige Testarten

Testart Merksatz Beispiel
Regressionstests "Wer vorwärts will, darf nicht zurückfallen." Nach einer Code-Änderung die gesamte Test-Suite laufen lassen, um sicherzustellen, dass alte Features noch funktionieren.
Smoke-Tests "Wo Rauch ist, ist Feuer. Kein Rauch? Gut, weiter geht's." Nach jedem Deployment kurz prüfen: Lädt die Startseite? Funktioniert der Login? Ja/Nein.
Manuelle / Explorative Tests "Vier Augen sehen mehr als zwei (Skripte)." Bewusst versuchen, die Anwendung kaputt zu machen: Sonderzeichen eingeben, schnell klicken etc.

Prinzipien & Guter Stil

Arrange, Act, Assert (AAA-Pattern)

"Vorbereiten, Ausführen, Überprüfen."

Beispiel: 1. Arrange: Erstelle einen Test-User. 2. Act: Rufe die Login-Funktion auf. 3. Assert: Prüfe, ob die Antwort ein gültiges Token enthält.

Teste nicht nur den "Happy Path"

"Jeder Plan überlebt nur bis zum ersten Kontakt mit dem User."

Beispiel: Was passiert, wenn ein Formular leer abgeschickt wird? Was, wenn die Datenbank fehlschlägt? Teste Fehlerfälle!

Tests sind lebende Dokumentation

"Guter Testcode erklärt, was der Code tun sollte."

Beispiel: Ein Test namens it('should not allow login with wrong password') ist selbsterklärend.
